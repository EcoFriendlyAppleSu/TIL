🍎 새로 알게된 사실 혹은 알고 있던 사실에대한 질문, 답변

🍏 테스트 코드는 명세(Application Design)을 테스트해야합니다.

❓ Test Code를 작성할 때 어느 수준으로 테스트를 작성해야할까요?

→ 작성하는 테스트 코드 또한 Production Code를 사용하는 여러 클라이언트 코드 중 하나일 뿐이고 그렇기에 OOP 원칙인 **정보 은닉**은 테스트 코드에게도 당연히 적용되어야지만, 우리가 구현하는 객체들은 자율성을 보장받을 수 있고, 따사ㅓ, 테스트 대상 입장에서 테스트 코드 또한 정보은익 대상이 되게 됩니다.

🍏 테스트 코드를 작성하기 위해선 최대한 결정적 코드이며 부수효과가 없어야합니다.

→ 비결정적 코드는 실행 시점마다 결과가 달라지는 코드를 이야기합니다. 반대로, 결정적인 코드는 매번 실행할 때마다 결과가 같은 것을 의미합니다.

❓ TDD를 하라는 말인가요?
→ 아닙니다. TDD는 실패하는 테스트를 먼저 작성하고 이를 성공시키는 Production Code를 작성하고 리팩토링하는 과정을 반복적으로 수행해 애플리케이션을 구현하는 과정입니다.

→ 여기서 이야기하는 테스트 코드 작성은 Production Code를 먼저 작성하고 이를 "검증"하는 것을 의미합니다.

🍊 아래와 같은 Cycle로 개발을 진행한다면 좋지 않을까라는 생각을 합니다.
1. Test Class를 만들고 해결해야 할 명세(요구사항)을 생성합니다.
2. Production Code를 작성해 요구사항을 풀어갑니다. 이 때, 최대한 결정적이며 부수효과가 없게 작성합니다.
3. 테스트 코드를 작성해 검증을 진행합니다.
